## 1. 引言

### 1.1 目的

本文档旨在明确 EAMS2026（Enterprise Asset & Approval Management System 2026）企业管理系统 v0.2.260605 版本的功能需求和非功能需求，为开发团队和利益相关者提供统一的需求基线。

### 1.2 范围

EAMS2026 是一个**局域网部署的 BS 架构企业管理系统**，覆盖以下业务领域：

- **基础管理**：部门、员工、用户、角色、权限、字典
- **考勤管理**：考勤记录、排班方案、打卡数据同步、考勤报表
- **消息系统**：站内消息收发、系统通知
- **数据管理**：数据权限控制、导入导出、仪表盘配置
- **系统监控**：操作日志审计

## 2. 总体描述

### 2.1 产品概述

EAMS2026 将现有多个 C# 程序（BS+CS）功能整合重构为一个统一的企业管理系统。采用**先跑通、再优化**的策略，基于 .NET 8 + Vue 3 + PostgreSQL 技术栈，支持局域网多用户并发访问。

**产品定位：**

- 面向中大型企业的内部管理系统
- 纯局域网部署，数据不出企业内网
- 模块化设计，可按需扩展

### 2.2 用户特征

| 用户角色 | 说明 | 典型操作 |
|----------|------|----------|
| 超级管理员 | 拥有系统所有权限 | 系统配置、权限分配、数据管理 |
| 系统管理员 | 管理基础数据和用户 | 部门/员工/用户/角色/权限管理 |
| 考勤管理员 | 管理考勤模块 | 同步打卡数据、维护排班方案、生成考勤报表 |
| 普通员工 | 使用系统基础功能 | 查看仪表盘、修改个人资料、收发消息 |
| 消息接收者 | 接收系统消息通知 | 查看收件箱、回复消息 |

### 2.3 运行环境

| 环境 | 说明 |
|------|------|
| 服务器操作系统 | Debian 12 (生产环境) / Windows 10+ (开发环境) |
| 应用服务器 | .NET 8 (Kestrel) |
| Web 服务器 | NGINX 1.22+ (反向代理 + 静态资源) |
| 数据库 | PostgreSQL 18.3 |
| 客户端 | 现代浏览器（Chrome/Firefox/Edge） |
| 部署方式 | 三节点分离部署（Nginx / API / PostgreSQL） |

### 2.4 设计约束

- 纯局域网部署，不依赖公网服务（钉钉数据同步除外）
- 基于 RBAC 模型的细粒度权限控制
- 所有 API 请求需携带 JWT 令牌认证
- 密码使用 BCrypt 算法哈希存储
- 关键操作记录审计日志

### 3.2 用例描述

#### 用例 1：用户登录

| 项目 | 内容 |
|------|------|
| 用例 ID | UC-001 |
| 参与者 | 所有用户 |
| 前置条件 | 用户已注册且账户状态正常 |
| 后置条件 | 系统返回 JWT 令牌，前端切换至主界面 |

**基本流程：**

1. 用户访问系统 URL，显示登录页面
2. 用户输入用户名和密码
3. 系统验证凭据，检查账户状态
4. 系统生成 JWT 令牌，返回用户信息、角色和权限列表
5. 前端存储令牌，跳转至仪表盘主页

**异常流程：**

- 用户名或密码错误 → 提示"用户名或密码错误"
- 账户已禁用 → 提示"账户已被禁用"
- 首次登录（强制修改密码）→ 跳转至修改密码页面

#### 用例 2：考勤数据同步

| 项目 | 内容 |
|------|------|
| 用例 ID | UC-002 |
| 参与者 | 考勤管理员 |
| 前置条件 | 用户拥有考勤管理权限 |

**基本流程：**

1. 用户进入考勤导入页面
2. 用户选择数据源（HWATT 或 钉钉）
3. 系统从数据源同步员工基础信息
4. 系统从数据源同步打卡记录
5. 系统自动匹配员工映射关系
6. 系统生成同步报告

#### 用例 3：考勤报表查询

| 项目 | 内容 |
|------|------|
| 用例 ID | UC-003 |
| 参与者 | 考勤管理员 |
| 前置条件 | 用户拥有考勤报表查看权限 |

**基本流程：**

1. 用户进入考勤报表页面
2. 用户选择员工和时间范围
3. 系统计算考勤汇总数据
4. 系统显示报表（正常出勤、迟到、早退、缺卡等统计）

### 3.3 功能需求列表（EARS 格式）

#### MUST（关键需求）

- **REQ-001**：当用户提交登录请求时，系统应验证用户名和密码，返回 JWT 令牌
- **REQ-002**：当用户访问受保护 API 时，系统应验证请求携带的 JWT 令牌
- **REQ-003**：当管理员创建/编辑用户时，系统应对密码进行 BCrypt 哈希存储
- **REQ-004**：系统应支持基于 RBAC 模型的细粒度权限控制
- **REQ-005**：当用户执行部门/员工/用户 CRUD 操作时，系统应记录操作日志
- **REQ-006**：系统应支持从 HWATT（SQL Server）同步考勤员工和打卡记录
- **REQ-007**：系统应支持从钉钉 API 同步考勤数据
- **REQ-008**：当用户查询考勤报表时，系统应计算指定日期范围内的考勤统计
- **REQ-009**：系统应支持部门、员工、用户、角色、权限的完整 CRUD 操作
- **REQ-010**：系统应支持站内消息的发送、接收和回复

#### SHOULD（重要需求）

- **REQ-011**：系统应支持字典类型的动态配置和管理
- **REQ-012**：当管理员配置仪表盘时，系统应允许自定义仪表盘组件
- **REQ-013**：系统应支持数据权限规则配置（按角色控制数据可见范围）
- **REQ-014**：系统应支持考勤员工与系统员工的手动/自动映射
- **REQ-015**：系统应支持打卡记录的批量导入和查询

#### COULD（期望需求）

- **REQ-016**：系统应支持考勤费用计算功能
- **REQ-017**：当用户导入数据时，系统应通过 SignalR 实时推送导入进度
- **REQ-018**：系统应支持考勤排班方案和班次管理
- **REQ-019**：系统应支持考勤节假日管理和日类型标记

## 4. 非功能需求

### 4.1 性能需求

| 编号 | 需求描述 | 指标 |
|------|----------|------|
| NFR-001 | API 响应时间 | 非报表页面 < 500ms，考勤报表 < 3s |
| NFR-002 | 同步效率 | 万级打卡记录同步 < 30s |
| NFR-003 | 并发用户数 | 支持 ≥ 100 用户同时在线 |

### 4.2 安全性需求

| 编号 | 需求描述 |
|------|----------|
| NFR-004 | 密码使用 BCrypt 哈希存储，不允许明文存储 |
| NFR-005 | JWT 令牌签名密钥长度 ≥ 32 字符 |
| NFR-006 | 所有 API 请求必须携带有效 JWT 令牌 |
| NFR-007 | 用户权限变更后，下一次请求即生效 |
| NFR-008 | 操作日志记录所有关键操作的详细上下文 |

### 4.3 可用性需求

| 编号 | 需求描述 |
|------|----------|
| NFR-009 | 前端使用 Element Plus 组件库，交互符合主流管理后台习惯 |
| NFR-010 | 关键操作（删除等）需二次确认 |
| NFR-011 | 表单输入进行前端和后端双重验证 |
| NFR-012 | 错误信息以友好方式展示给用户 |

### 4.4 可维护性需求

| 编号 | 需求描述 |
|------|----------|
| NFR-013 | 采用分层架构（API / Application / Domain / Infrastructure） |
| NFR-014 | 数据库迁移使用 SQL 脚本维护，版本控制 |
| NFR-015 | 日志使用 Serilog，支持按天切割和保留策略 |
| NFR-016 | 前后端分离，通过 REST API 通信 |

## 5. 用户故事

### 5.1 高优先级（Must have）

- 作为**用户**，我想要登录系统，以便于访问授权的功能模块
- 作为**管理员**，我想要管理部门和员工信息，以便于建立组织架构
- 作为**管理员**，我想要创建系统和分配角色权限，以便于控制用户访问
- 作为**考勤管理员**，我想要同步考勤机打卡数据，以便于生成考勤记录
- 作为**考勤管理员**，我想要查看考勤报表，以便于统计员工出勤情况

### 5.2 中优先级（Should have）

- 作为**管理员**，我想要查看操作日志，以便于审计系统使用情况
- 作为**用户**，我想要修改个人资料和密码，以便于维护个人信息
- 作为**管理员**，我想要配置数据权限规则，以便于控制不同角色的数据范围

### 5.3 低优先级（Could have）

- 作为**管理员**，我想要自定义仪表盘组件，以便于展示关注的业务数据
- 作为**管理员**，我想要配置考勤费用计算规则，以便于自动化薪资计算

## 6. 术语表

|------|------|
| RBAC | 基于角色的访问控制模型，用户通过角色关联权限 |

| 数据权限 | 按角色控制数据可见范围的机制 |

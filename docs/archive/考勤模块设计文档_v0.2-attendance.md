## 一、需求分析

### 1.1 业务背景
根据现有考勤功能代码分析，本模块需要实现以下核心业务：

| 业务需求 | 描述 | 来源文件 |
|---------|------|----------|
| 考勤数据同步 | 从考勤机（SQL Server 2012 HWATT数据库）和钉钉API同步考勤数据 | `AttendanceBLL.cs`, `Attendance2_proc.sql` |
| 打卡记录查询 | 按员工、日期范围筛选考勤记录 | `VAttendanceReport/Index.razor` |
| 考勤异常处理 | 自动识别迟到/早退/缺卡，支持手动标记事件 | `Attendance2EventDeclared.cs` |
| 考勤统计报表 | 月度出勤统计、费用计算 | `VAttendanceReport.cs` |
| 排班计划管理 | 配置班次、排班标准时间、日期类型 | `Attendance2PlanTime.cs`, `Attendance2SchemeClass.cs` |
| 假日调休管理 | 管理节假日和调休日期 | `Attendance2Holiday.cs` |
| 考勤打印 | 生成考勤报表PDF，支持打印输出 | `PrintView.razor` |
| 考勤计算 | 自动排班、时长计算、费用计算 | `Attendance2_proc.sql` |

### 1.2 双数据源说明

考勤管理的数据来源分为两部分：

| 数据源 | 适用范围 | 数据内容 | 获取方式 |
|--------|----------|----------|----------|
| **HWATT考勤机** (SQL Server) | 储运部打卡设备，其他部门员工也可在此打卡 | 员工信息、打卡记录 | 直连SQL Server只读查询 |
| **钉钉API** | 其他部门使用钉钉打卡，储运部员工出差也可使用 | 员工信息、打卡记录 | 调用钉钉开放平台API |

**核心设计原则：**
- 打卡记录 **按来源全量导入**，导入时不按员工部门过滤
- 考勤计算时，对每个员工每天 **合并两个来源的打卡记录**，取有效值统一计算
- `attendance_employees` 表中的 `source` 字段仅表示员工的默认排班归属，**不决定**打卡数据的来源

**典型场景：**
| 场景 | HWATT打卡记录 | 钉钉打卡记录 | 计算结果 |
|------|--------------|-------------|---------|
| 财务部员工去储运部HWATT打卡 | ✅ 有记录 | ❌ 无 | ✅ 正确计算 |
| 储运部员工出差用钉钉打卡 | ❌ 无 | ✅ 有记录 | ✅ 正确计算 |
| 某员工在两个系统都打了卡 | ✅ | ✅ | ✅ 合并取最早/最晚 |

### 1.3 外部数据库依赖

由于HWATT数据库（考勤机数据库）无法迁移到PostgreSQL，系统需要同时连接两个数据库：

| 数据库 | 类型 | 用途 | 连接方式 |
|--------|------|------|----------|
| PostgreSQL | 主数据库 | 存储业务数据、考勤记录、同步缓存 | 默认连接 |
| SQL Server 2012 (HWATT) | 外部数据库 | 读取员工信息和打卡记录 | 只读连接 |

**HWATT数据库表结构：**

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `KQZ_Employee` | 员工信息表 | `EmployeeID`, `EmployeeName`, `BrchID` |
| `KQZ_Card` | 打卡记录表 | `EmployeeID`, `CardTime`, `CardTypeID`, `DevID` |

**同步策略（双数据源）：**

```
┌─────────────────────────────────────────────────────────────────────┐
│                       考勤数据导入总览                               │
│                                                                     │
│  HWATT全量导入 ──► hwatt_employees (原始缓存)                       │
│                  ──► hwatt_card_records (原始缓存)                   │
│                                                                     │
│  钉钉全量导入 ──► dingtalk_employees (原始缓存，未来)               │
│    (未来实现)    ──► dingtalk_card_records (原始缓存，未来)          │
│                                                                     │
│  考勤计算 ──► 读取两个来源的打卡记录，合并处理                      │
│            ──► 保存到 attendance_records (统一计算结果)             │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 数据源架构设计（V0.2新增）

#### 2.3.1 核心概念

```
考勤员工 (attendance_employees)
├── employee_ref_id ────► employees(id)    (HR系统员工)
├── source: 'hwatt' | 'dingtalk'           (默认排班归属，不决定打卡数据来源)
├── hwatt_employee_id                       (HWATT原始ID，仅source='hwatt'时使用)

打卡数据通道:
├── HWATT通道: KQZ_Card ──同步──► hwatt_card_records ──┐
│                                                        ├──► attendance_records
├── 钉钉通道: 钉钉API ──同步──► dingtalk_card_records ──┘    (合并后计算结果)
```

#### 2.3.2 数据流

```
┌──────────┐   全量同步    ┌────────────────┐
│ HWATT DB │─────────────►│ hwatt_card_    │
│ KQZ_Card │              │ records        │
└──────────┘              └───────┬────────┘
                                  │
                                  │ 按员工+日期查询
                                  ▼
┌──────────┐   全量同步    ┌────────────────┐     ┌──────────────────┐
│ 钉钉API  │─────────────►│ dingtalk_card_  │────►│  考勤计算服务     │
│ (未来)    │   (未来)      │ records (未来)   │     │  合并两个来源     │
└──────────┘              └────────────────┘     │  写入attendance  │
                                                  │  _records        │
                                                  └──────────────────┘
```

#### 2.3.3 同步与导入的职责分离

| 操作 | 说明 | 触发方式 | 数据流向 |
|------|------|----------|----------|
| **HWATT员工同步** | 从HWATT读取员工列表，按姓名匹配系统HR员工，写入考勤员工表 | 手动/定时 | KQZ_Employee → hwatt_employees → attendance_employees |
| **HWATT打卡同步** | 从HWATT读取指定日期范围的打卡记录，缓存到本地 | 手动/定时 | KQZ_Card → hwatt_card_records |
| **钉钉打卡同步** | 从钉钉API获取打卡记录，缓存到本地（预留） | 手动/定时 | 钉钉API → dingtalk_card_records（未来） |
| **考勤计算导入** | 按员工+日期，从所有来源取打卡记录，合并计算后写入结果表 | 手动 | hwatt_card_records + dingtalk_card_records → attendance_records |

### 3.3 数据库表结构

#### 3.3.1 attendance_employees（考勤员工表 - V0.2新增字段）

```sql
CREATE TABLE attendance_employees (
    id BIGSERIAL PRIMARY KEY,
    employee_name VARCHAR(100) NOT NULL,
    employee_id BIGINT NOT NULL UNIQUE,
    employee_ref_id BIGINT REFERENCES employees(id),
    source VARCHAR(20) DEFAULT 'hwatt',
    hwatt_employee_id BIGINT,
    status BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW(),
    created_by BIGINT DEFAULT 0,
    updated_at TIMESTAMP DEFAULT NOW(),
    updated_by BIGINT DEFAULT 0,
    is_deleted BOOLEAN DEFAULT FALSE
);

COMMENT ON COLUMN attendance_employees.source IS '数据源: hwatt=HWATT考勤机, dingtalk=钉钉API';
COMMENT ON COLUMN attendance_employees.employee_ref_id IS '关联HR系统employees表';
COMMENT ON COLUMN attendance_employees.hwatt_employee_id IS 'HWATT考勤机原始员工ID';
```

## 四、考勤计算算法设计（从SQL存储过程迁移）

### 4.1 日期类型定义

| DayTypeID | 类型名称 | 说明 |
|-----------|----------|------|
| 1 | WorkDay | 工作日 |
| 2 | Duty | 值班 |
| 3 | Over | 加班 |
| 4 | DayOff | 假日 |
| 5 | Holiday | 节日 |
| 6 | Absence | 缺勤 |
| 7 | Leave | 请假 |

### 4.2 核心计算流程

#### 4.2.1 考勤记录保存（Attendance_SaveAttendanceRecord）

**输入参数：**
- `empId`: 员工ID
- `sDate`: 考勤日期
- `cardBegin`: 上班打卡时间
- `cardEnd`: 下班打卡时间

**处理逻辑：**
1. 查询员工当日是否已有考勤记录
2. 如果不存在，插入新记录
3. 如果存在，更新记录：
   - 上班时间取较早的时间（打卡时间 vs 已有记录）
   - 下班时间取较晚的时间（打卡时间 vs 已有记录）

#### 4.2.2 单日排班计算（Attendance_scheduleOneDay）

**输入参数：**
- `eid`: 员工ID
- `scheduleDate`: 排班日期

**处理逻辑：**
1. **获取上日考勤信息**：查询上一日的周期序号(`periodNo`)和排班计划ID(`planid`)
2. **计算周期序号**：`period2 = (prev_period2 + 1) % periods`，若为0则取`periods`
3. **获取班次标准时间**：根据周期序号查询当日的上班时间(`bTime`)、下班时间(`eTime`)
4. **检查生效日**：如果当日是新排班计划的生效日，重新获取排班信息
5. **检查节日**：查询假日表，若为节日则更新日期类型和时间
6. **周期复位**：如果是特殊计划(planid=11,12)且为周六，执行周期复位
7. **保存排班记录**：插入或更新考勤记录

#### 4.2.3 周期复位算法（Attendance_resetSchedulePeriod）

**触发条件：**
- 当前计划为节日计划(planid=11,12)且日期为周六

**处理逻辑：**
1. 获取班次周期数(`periods`)和班次内周期数(`classPeriods`)
2. 计算复位后的周期序号：`period = (@periods + (@period - @classPeriods)) % @periods`

#### 4.2.4 单日考勤计算（Attendance_reCalculaorOneDay）

**输入参数：**
- `eid`: 员工ID
- `attDate`: 考勤日期

**处理逻辑：**
1. **查询考勤记录**：获取当日的打卡时间、标准时间、日期类型等信息
2. **计算时长偏移**：
   - `bOverTime = DATEDIFF(mi, bAttTime, bTime)` - 上班偏移（负数表示迟到）
   - `eOverTime = DATEDIFF(mi, eTime, eAttTime)` - 下班偏移（正数表示加班）
   - 特殊计划(planid=7,8,12)：`bOverTime=0`, `eOverTime = DATEDIFF(mi, bAttTime, eAttTime)`
3. **费用计算**：根据`dayTypeId`和偏移量从`attendance_fee_calculators`表查询费用
4. **更新记录**：更新`bOffset`, `eOffset`, `bOffsetFee`, `eOffsetFee`字段

#### 4.2.5 从HWATT导入考勤记录（V0.2重构）

**输入参数：**
- `empId`: 员工ID
- `beginDate`: 开始日期
- `endDate`: 结束日期

**处理逻辑（V0.2双数据源）：**
1. 遍历日期范围
2. 从`hwatt_card_records`（本地缓存）查询当日打卡记录
3. 从`dingtalk_card_records`（未来）查询当日打卡记录
4. 合并两个来源的记录，取最早/最晚时间作为上班/下班打卡时间
5. 调用`Attendance_SaveAttendanceRecord`保存合并后的记录

**说明**：V0.2将直接从SQL Server查询改为从本地同步表查询，提升性能并解耦。

#### 4.2.6 全员考勤计算（Attendance_doAllEmployeeAttendance）

**输入参数：**
- `doDate`: 开始日期
- `endDate`: 结束日期（默认当月最后一天）

**处理逻辑：**
1. 遍历所有有效员工（从`attendance_employees`查询）
2. 对每个员工，遍历日期范围
3. 对于每个员工每天：
   a. 从`hwatt_card_records`取打卡记录
   b. 从`dingtalk_card_records`（未来）取打卡记录
   c. 合并后执行：
      - `Attendance_SaveAttendanceRecord` - 保存记录
      - `Attendance_scheduleOneDay` - 自动排班
      - `Attendance_reCalculaorOneDay` - 计算时长和费用

### 6.2 考勤报告查询流程

```

     │                    │                          │

CREATE VIEW v_attendance_employee AS
SELECT 
    id,
    employee_name,
    employee_id,
    employee_ref_id,
    source,
    hwatt_employee_id
FROM attendance_employees
WHERE is_deleted = FALSE AND status = TRUE;
```

**v_attendance_report 视图（考勤报表视图）**

```sql
CREATE VIEW v_attendance_report AS
SELECT 
    r.id AS record_id,
    r.employee_id,
    e.employee_name,
    r.s_date,
    p.description,
    r.b_att_time,
    r.b_offset,
    r.e_att_time,
    r.e_offset,
    r.b_offset_fee,
    r.e_offset_fee,
    (SELECT string_agg(
        CASE WHEN ed.is_begin_time THEN '早:' ELSE '晚:' END || ed.event_description, 
        ';' ORDER BY ed.is_begin_time
    ) FROM attendance_event_declareds ed WHERE ed.record_id = r.id) AS event,
    (SELECT MAX(ed.fee) FROM attendance_event_declareds ed WHERE ed.record_id = r.id AND ed.fee > 0) AS fee
FROM attendance_records r
LEFT JOIN attendance_plan_times p ON r.plan_id = p.id
LEFT JOIN v_attendance_employee e ON r.employee_id = e.id
WHERE r.is_deleted = FALSE;
```

### 11.2 核心算法伪代码

```
// 考勤计算 - 双数据源合并
for each employee in attendance_employees:
    for each date in dateRange:
        // 从两个来源获取打卡记录
        hwattRecords = hwatt_card_records[employeeId, date]
        dingtalkRecords = dingtalk_card_records[employeeId, date]  // 未来
        
        // 合并取最早/最晚
        allRecords = hwattRecords ∪ dingtalkRecords
        if allRecords is empty: continue
        
        cardBegin = MIN(allRecords.CardTime)
        cardEnd   = MAX(allRecords.CardTime)
        
        // 保存合并后的记录
        SaveAttendanceRecord(employeeId, date, cardBegin, cardEnd)
        ScheduleOneDay(employeeId, date)
        ReCalculateOneDay(employeeId, date)
```

### 11.3 注意事项

| 风险点 | 说明 | 解决方案 |
|--------|------|----------|
| HWATT员工名与系统员工名不一致 | HWATT员工名可能包含特殊字符或简写 | 同步时记录匹配日志，支持手动匹配 |
| 同一员工在HWATT和钉钉都有记录 | 两个来源的打卡时间可能重叠或矛盾 | 合并取最早/最晚，保留source标记 |
| HWATT数据库连接超时 | 大量打卡记录查询可能超时 | 分批同步（按日期范围），增加重试机制 |
| 钉钉API限流 | 钉钉API有调用频率限制 | 实现请求队列，控制并发数（预留） |
| 同步数据一致性 | 同步过程中可能部分失败 | 使用同步日志表记录每次操作，支持断点续传 |

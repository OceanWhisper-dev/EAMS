## 2. 后端架构设计

### 2.1 分层架构

#### 2.1.1 API 层（EAMS2026.Api）

**路由结构：**

```
/api/auth/*             → AuthController（登录、个人信息、修改密码）
/api/department/*       → DepartmentController（部门树 CRUD）
/api/employee/*         → EmployeeController（员工信息 CRUD）
/api/user/*             → UserController（系统用户 CRUD）
/api/role/*             → RoleController（角色 CRUD）
/api/permission/*       → PermissionController（权限树 CRUD）
/api/dict/*             → DictController（字典类型和字典项 CRUD）
/api/attendance/*       → AttendanceController（考勤管理全功能）
/api/message/*          → MessageController（站内消息收/发/管理）
/api/dashboard/*        → DashboardController（仪表盘数据聚合）
/api/data-permission/*  → DataPermissionController（数据权限规则管理）
/api/operation-log/*    → OperationLogController（操作日志查询）
/api/import-export/*    → ImportExportController（员工导入导出）
/api/print/*            → PrintController（考勤表打印）
/api/reports/*          → ReportController（报表设计/执行/导出）
/hubs/import            → ImportHub（导入进度实时推送，SignalR）
```

**外部服务集成：**
- `HwattImportService`：从 HWATT 数据库同步考勤数据

**权限框架：**
- `PermissionHandler` + `PermissionRequirement`：基于角色-权限的授权策略
- `RequirePermissionAttribute`：权限声明式过滤 Attribute

### 2.2 认证与授权

#### 2.2.1 认证流程

```
客户端                         服务端
  │                             │
  │  POST /api/auth/login       │
  │  { username, password }     │
  │ ─────────────────────────►  │
  │                             │ BCrypt 验证密码
  │                             │ 生成 JWT Token
  │  { token, roles, ... }      │ (有效期 1440 分钟)
  │ ◄─────────────────────────  │
  │                             │
  │  GET /api/xxx               │
  │  Authorization: Bearer JWT  │
  │ ─────────────────────────►  │
  │                             │ JWT 验证 + 权限检查
  │  { success, data }          │
  │ ◄─────────────────────────  │
```

| `data-permission` | 数据权限管理权限 |

#### 2.2.3 数据权限

基于角色的数据权限规则，支持以下范围：

| 数据范围 | 说明 |
|----------|------|
| ALL | 所有数据 |
| DEPARTMENT | 本部门及子部门 |
| SELF_DEPT | 仅本部门 |
| SELF | 仅本人 |

### 2.3 统一的 API 响应格式

```json
// 成功响应
{ "success": true, "data": { ... }, "message": "操作成功" }

// 失败响应
{ "success": false, "message": "错误描述" }
```

所有 Controller 继承 `BaseController`，通过 `Success()` / `Fail()` 辅助方法生成统一响应。

### 2.4 异常处理

`ExceptionMiddleware` 全局捕获未处理异常：
- `UnauthorizedAccessException` → 401
- `KeyNotFoundException` → 404
- 其他 → 500 + 日志记录

### 2.5 操作日志

`OperationLogMiddleware` 自动记录所有非 GET 请求的操作日志，记录内容包括：
- 操作人、IP、URL、HTTP 方法
- 请求参数（脱敏）
- 执行耗时
- 操作结果

## 3. 报表模块设计

### 3.1 模块概述

报表模块是 EAMS2026 的核心功能模块，支持自定义 SQL 报表的设计、执行、导出和可视化。

**功能矩阵：**

| 功能 | 说明 | 状态 |
|------|------|------|
| 报表分类管理 | 树形分类，支持无限层级 | ✅ |
| 报表设计器 | 在线编辑 SQL、配置字段/过滤/排序/图表 | ✅ |
| 报表执行 | 参数化查询、服务端分页、服务端排序 | ✅ |
| 数据导出 | Excel 导出（含当前页/全部） | ✅ |
| 图表展示 | ECharts 集成（柱状/折线/饼图） | ✅ |
| 权限控制 | 报表级权限 + 字段级权限 | ✅ |
| 数据源切换 | 主库(PostgreSQL) / ERP(SQL Server) | ✅ |
| 收藏/执行日志 | 常用报表收藏、执行历史记录 | ✅ |
| 透视表 | PivotTable.js 集成，行/列/值拖拽 | ✅ |
| 销售员地区映射 | 地图可视化，ECharts 地图 | ✅ |

### 3.2 核心流程

```
报表执行流程：

用户点击执行
    │
    ▼
ReportController.ExecuteReport(id, request)
    │
    ▼
ReportService.ExecuteReportAsync(id, request, userId)
    │
    ├─ 1. 验证权限（报表级 + 字段级）
    ├─ 2. 参数注入（@param → 参数值）
    ├─ 3. 服务端排序（Sort 参数覆盖默认 ORDER BY）
    ├─ 4. 分页查询（COUNT + 分页 SQL）
    ├─ 5. 字段类型映射（date/money → 前端格式化）
    └─ 6. 返回结果（Columns + Rows + Pagination + ExecutionInfo）

前端渲染：
    Columns → el-table-column 定义
    Rows → 表格数据
    字段类型 date/money → formatDateTimeSafe / formatMoney
```

### 3.3 数据库表设计

| 表 | 前缀 | 说明 |
|----|------|------|
| rpt_category | 报表分类 | 树形结构，支持父级关联 |
| rpt_report | 报表主表 | 存储 SQL、数据源、状态 |
| rpt_field | 报表字段 | 字段名/标题/类型/宽度/对齐/汇总 |
| rpt_filter | 报表过滤条件 | 参数化输入控件定义 |
| rpt_sort | 报表默认排序 | 字段 + 方向 |
| rpt_chart | 报表图表 | ECharts 配置 JSON |
| rpt_permission | 报表权限 | 角色-报表可见性 |
| rpt_field_permission | 字段权限 | 角色-字段可见性 |
| rpt_data_source | 报表数据源 | 备用连接字符串配置 |
| rpt_bookmark | 报表收藏 | 用户-报表收藏 |
| rpt_execution_log | 执行日志 | 查询耗时/行数 |

### 3.4 防 SQL 注入

所有动态查询通过 Dapper 参数化执行，排序字段通过 `SanitizeParamName()` 白名单过滤：

```csharp
private static string SanitizeParamName(string name)
{
    // 只允许字母、数字、下划线，防止注入
    return Regex.Replace(name ?? "", "[^a-zA-Z0-9_]", "");
}
```

### 3.5 排序机制

支持前端传递排序参数覆盖 SQL 默认排序：

```
请求示例：
POST /api/reports/{id}/execute
{
    "sort": [{ "field": "amount", "direction": "desc" }],
    "pagination": { "page": 1, "pageSize": 50 }
}

后端处理：
1. 移除原始 SQL 中的 ORDER BY
2. 追加新的 ORDER BY（已清洗的字段名）
3. COUNT 查询中移除 ORDER BY（优化性能）
```

### 3.6 前端渲染

前端使用 Element Plus `el-table` 渲染报表结果，支持：
- 列宽、对齐、格式化（date/money 类型自动格式化）
- 列头排序（触发服务端排序）
- 汇总行（sum/avg/count/max/min）
- 行单击查看明细

## 4. 考勤模块设计

### 4.1 模块概述

考勤管理模块支持多数据源考勤数据采集、排班管理、考勤计算、报表统计。

**功能矩阵：**

| 功能 | 说明 | 数据源 |
|------|------|--------|
| 考勤员工映射 | HWATT/钉钉员工 ↔ 系统员工 | HWATT SQL Server / 钉钉 API |
| 打卡记录同步 | 定时/手动同步打卡记录 | HWATT / 钉钉 |
| 排班管理 | 班次方案定义、员工排班 | 系统数据库 |
| 考勤计算 | 自动识别迟到/早退/缺卡/加班 | 系统计算引擎 |
| 节假日管理 | 法定节假日/公司假日配置 | 系统数据库 |
| 事件声明 | 员工加班/请假/出差/调休申请 | 系统数据库 |
| 费用计算 | 根据考勤结果自动计算费用 | 系统计算引擎 |
| 数据导入 | 从遗留系统导入历史考勤数据 | Excel / CSV |
| 考勤报表 | 月度出勤统计 | 系统数据库 |

### 4.2 数据同步架构

```
HWATT (SQL Server)
    │  HwattImportService (定时/手动)
    ▼
考勤员工表 → 员工映射表
打卡记录表
    │
    ├── 钉钉 API
    │   DingTalkService (定时/手动)
    ▼
考勤员工表 → 员工映射表
打卡记录表
    │
    ▼
AttendanceCalculationService
    │
    ├── 排班方案 + 节假日
    ├── 事件声明
    ▼
考勤记录表 → 考勤报表
    │
    ▼
AttendanceFeeCalculator → 费用计算结果
```

```
    │

    │
    ▼
Axios 响应拦截器 (错误处理)
    │
    ▼

```

```

    │
```

## 8. 安全设计

### 8.1 认证安全

- 密码使用 BCrypt 哈希存储
- JWT Token 有效期 1440 分钟（可配置）
- JWT SecretKey 必须从配置文件读取，不提供硬编码回退
- Token 过期后自动跳转登录页

### 8.2 授权安全

- 基于角色的权限控制（RBAC）
- 前端 `v-permission` 指令：按钮级可见性控制
- 后端 `[Authorize(Policy=)]`：接口级访问控制
- 数据权限过滤器：行级数据可见性

### 8.3 数据安全

- 参数化 SQL 查询（Dapper）：防止 SQL 注入
- 动态排序字段名白名单清洗（`SanitizeParamName`）
- 软删除（`is_deleted` 标志）：所有实体继承 `BaseEntity`
- CORS 策略：生产环境限定来源

### 8.4 审计安全

- 操作日志中间件自动记录所有写操作
- 日志包含操作人、时间、IP、操作内容
- 日志保留 30 天（文件滚动策略）

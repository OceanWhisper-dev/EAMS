```

/api/employee/*       → EmployeeController（员工信息 CRUD）
/api/user/*           → UserController（系统用户 CRUD）
/api/role/*           → RoleController（角色 CRUD）
/api/permission/*     → PermissionController（权限树 CRUD）

### 2.2 认证与授权

#### 2.2.1 JWT 认证流程

```
登录请求 (POST /api/auth/login)
    │
    ▼
验证用户名 + 密码 (BCrypt)
    │
    ▼
获取用户角色和权限列表
    │
    ▼
生成 JWT Token (包含: userId, roles, permissions)
    │
    ▼
返回 LoginResponse { token, username, roles, permissions, ... }
    │
    ▼
前端存储令牌 (Pinia + localStorage)
    │
    ▼
后续请求携带 Authorization: Bearer {token}
```

#### 2.2.2 权限模型（RBAC）

```
用户 (User) ────多对多──── 角色 (Role) ────多对多──── 权限 (Permission)
                              │
                              ├── super_admin (超级管理员 - 拥有所有权限)
                              ├── admin (系统管理员)
                              └── user (普通用户)
```

**权限编码：**

| 模块 | 权限编码 |
|------|----------|
| 部门管理 | department |
| 员工管理 | employee |
| 用户管理 | user |

                                         │
                                         ▼
                                    考勤报表
                                    (正常/迟到/早退/缺卡/费用)
```

## 5. 安全设计

### 5.1 传输安全

- 内网部署，不强制 HTTPS
- JWT 令牌通过 HTTP Header 传输
- SignalR WebSocket 通过 Query String 携带令牌

### 5.2 存储安全

- 密码使用 BCrypt 哈希存储（自动加盐）
- JWT SecretKey 必须通过配置文件设置，无默认值
- 操作日志记录关键操作上下文

### 5.3 权限安全

- 所有 Controller 默认需要 JWT 认证（除 `/api/auth/login`）
- 模块级权限通过 `[Authorize(Policy = "...")]` 控制
- 支持数据级权限控制（按角色限定数据范围）

## 拆分可行性分析

基于对 v0.2.260605 代码的深入理解，以下是拆分方案分析。

---

### 一、当前耦合现状

**现状：单体解决方案，但代码层面已有一定的分层**

| 维度 | 现状 | 耦合程度 |
|------|------|----------|
| 项目结构 | 单一 `.sln`，4 个项目 + 1 前端 | 🔴 单体 |
| 命名空间 | `EAMS2026.*` 统一前缀 | 🟡 可重命名 |
| API 路由 | 所有 Controller 都在 `EAMS2026.Api` | 🔴 单体 |
| 数据库 | 统一 `eams2026` 数据库，共享 `employees`/`departments` 等表 | 🟡 可分离 |
| 前端路由 | 考勤和系统管理在同一个 `router/index.ts` | 🟡 可通过 lazy load 拆分 |
| 认证/权限 | 考勤模块使用框架的 JWT + RBAC | 🟢 合理依赖 |
| 基类依赖 | `AttendanceController` 继承 `BaseController`（提供 `GetUserId()` 等） | 🟢 合理依赖 |

### 二、拆分方案

#### 方案 A：解决方案内多项目拆分（推荐起步）

```
EAMS2026.Framework.sln                  ← 基础框架
├── EAMS2026.Framework.Api              ← 框架 API（登录/部门/员工/用户/角色/权限/字典/日志）
├── EAMS2026.Framework.Application      ← 框架业务逻辑
├── EAMS2026.Framework.Domain           ← 框架实体（部门/员工/用户/角色/权限等）
├── EAMS2026.Framework.Infrastructure   ← 框架数据访问 + JWT + SignalR
└── EAMS2026.Framework.Web              ← 框架前端（系统管理 UI + 登录 + 布局）

EAMS2026.Attendance.sln                 ← 考勤模块
├── EAMS2026.Attendance.Api             ← 考勤 API（引用框架的 Api）
├── EAMS2026.Attendance.Application     ← 考勤业务逻辑（引用框架的 Application）
├── EAMS2026.Attendance.Domain          ← 考勤实体（独立，可引用框架实体）
├── EAMS2026.Attendance.Infrastructure  ← 考勤数据访问（引用框架的 Infrastructure）
└── EAMS2026.Attendance.Web             ← 考勤前端（引用框架的前端组件库）

EAMS2026.sln                            ← 主解决方案（引用两个子方案）
```

**复杂度评估：**

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 后端代码拆分 | ⭐⭐⭐ 中 | 考勤的 Controller/Service/Domain 已按命名空间隔离，提取为独立项目 **1-2 天** |
| 数据库拆分 | ⭐⭐ 低 | 考勤表有独立前缀 `attendance_*`，只需将迁移脚本分离 **0.5 天** |
| 前端拆分 | ⭐⭐⭐⭐ 高 | Vue 3 模块联邦/Vite 插件加载需要额外工程配置 **2-3 天** |
| 依赖管理 | ⭐⭐⭐ 中 | 框架 NuGet 包需要发布到本地源或项目引用 **1 天** |

#### 方案 B：模块化插件架构（终极目标）

```
EAMS2026.Host (主机)
├── EAMS2026.Abstractions              ← 模块接口定义（IModule, IModuleStartup）
├── ModuleLoader                       ← 动态加载模块的程序集
│
├── EAMS2026.Framework (内建模块)      ← 作为默认模块
└── EAMS2026.Attendance (独立模块)     ← 作为插件，可独立开发/部署
```

**特点：**
- 考勤模块编译为独立 DLL，通过 DI 注入主机
- 前端使用 **Module Federation（Webpack 5）** 或 **Vite 插件** 动态加载
- 最灵活，但当前阶段过度设计

**复杂度评估：⭐⭐⭐⭐⭐ 很高**（需要大量重构，不适合当前阶段）

### 三、关键难点分析

#### 难点 1：前端模块化加载

```
当前：单一 SPA，所有页面在同一个 Vite 项目中
                     ↓
目标：框架提供「插槽」，考勤页面可插拔
```

**可选方案对比：**

| 方案 | 复杂度 | 说明 |
|------|--------|------|
| **Monorepo + 分包** | ⭐⭐ | 同一个仓库下用 pnpm workspace 管理，考勤作为独立 package |
| **Vite Module Federation** | ⭐⭐⭐⭐ | 运行时动态加载远程模块，需要 @originjs/vite-plugin-federation |
| **iframe 嵌入** | ⭐ | 最简单但交互割裂，不推荐 |

#### 难点 2：数据库层面

```
当前：单库单 schema，表混在一起
eams2026 (database)
├── employees           ← 框架
├── departments         ← 框架
├── users               ← 框架
├── roles               ← 框架
├── attendance_employees  ← 考勤
├── attendance_records    ← 考勤
└── ...
```

**拆分方式：**
- **逻辑分区（推荐）**：保持同一个库，用 schema 区分（如 `framework.*` 和 `attendance.*`）
- **物理分区**：两个独立数据库，考勤通过 API 或 DB Link 引用框架数据

> 考勤模块的 `attendance_employees.employee_id` 引用了框架的 `employees.id`，需要处理外键约束。

#### 难点 3：认证与权限共享

```
考勤 API → 使用框架 JWT 认证
         → 使用框架 RBAC 权限策略 (authorize policy = "attendance")
         → 使用框架 UserId 获取当前用户
         → 使用框架 OperationLog 记录审计日志
```

**解决方案：**
- 方案 A 下：考勤项目直接引用框架项目，这些共享无需改动
- 方案 B 下：需要定义 `IAuthContext` 接口，由主机注入

#### 难点 4：数据库迁移版本管理

```
当前：001-021 混合了框架和考勤的迁移脚本
001_InitialSchema.sql          ← 框架
008_AttendanceTables.sql       ← 考勤
009_HwattSyncTables.sql        ← 考勤
```

**需重命名/分离为：**
```
framework/001_InitialSchema.sql
framework/002_Messages.sql
attendance/001_AttendanceTables.sql
attendance/002_HwattSyncTables.sql
```

### 四、推荐实施路线

```
阶段 1（1-2 周）：解决方案内多项目拆分
├── 后端
│   ├── 新建 EAMS2026.Framework.* 四个项目（从当前复制）
│   ├── 新建 EAMS2026.Attendance.* 四个项目（从当前复制）
│   ├── 考门模块引用框架项目（认证/基类/共享实体）
│   ├── 分离数据库迁移脚本
│   └── 调整 namespace
│
├── 前端
│   └── 用 pnpm monorepo 拆分
│       ├── packages/framework/ (布局/登录/系统管理)
│       └── packages/attendance/ (考勤页面)
│
└── 验证：整体能编译运行

阶段 2（0.5 天）：框架 NuGet 包化
├── 将 Framework 打包为 NuGet（发布到本地 Nexus 或 nuget.config 本地源）
├── 考勤模块通过 NuGet 引用框架
├── 新模块通过同样方式引用框架
└── 验证：考勤模块可独立编译

阶段 3（可选）：前端模块化
├── 配置 Vite Module Federation
├── 考勤页面在运行时动态加载
└── 验证：框架不部署考勤也能正常运行
```

### 五、可行性结论

| 评估项 | 结论 |
|--------|------|
| **可行性** | ✅ **可行**，当前架构已有良好的分层基础 |
| **推荐方案** | **方案 A**（解决方案内多项目拆分） |
| **预估工时** | **约 2 周**（含测试验证） |
| **主要风险** | 前端模块化加载、数据库迁移版本管理 |
| **最大价值** | 考勤模块可独立演进，新模块可从 Framework 快速构建 |
| **实施建议** | **阶段 1 完成后即可投入实用**，阶段 2/3 视需要逐步推进 |

**一句话结论：** 当前代码的命名空间和分层已经为拆分做了良好铺垫，实施方案 A 风险可控、收益明确，建议直接进入实施阶段。

# 换班申请功能设计方案

> 文档版本: v1.0  
> 创建日期: 2026-05-30  
> 状态: 待实现

---

## 一、需求概述

员工之间存在换班需求时（例如员工A和员工B在某一天互换班次），需要提供换班申请、审批流程，并在考勤计算时按换班后的排班执行。

---

## 二、现有排班机制回顾

### 2.1 核心数据表

**attendance_employee_ref_scheme_classes** — 员工班次关联表

| 字段 | 类型 | 含义 |
|------|------|------|
| employee_id | BIGINT | 考勤员工ID（对应打卡机里的ID） |
| class_id | BIGINT | 班次ID（如"日常"、"周六仓库值班"等） |
| period_no | INT | 周期序号（班次轮换的起始位置） |
| eff_date | DATE | 生效日期 |
| exp_date | DATE? | 失效日期（NULL表示永久） |

### 2.2 排班与考勤计算流程

考勤计算核心在 `AttendanceCalculationService.cs` 中，分三步：

```
ScheduleOneDayAsync() → 排班
    1. GetEmployeeActiveSchemeAsync()   → 查找员工在指定日期生效的排班方案
    2. 根据 daysSinceEff 计算 period_no → 轮转周期计算
    3. 写入 attendance_records（class_id, plan_id, period_no, b_time, e_time, day_type_id）

ReCalculateOneDayAsync() → 考勤计算
    基于 attendance_records 中已排好的班次信息，对比打卡时间计算迟到早退和费用
```

### 2.3 关键规律

- **班次排班** 和 **考勤计算** 是分离的两个步骤
- 排班决定"应该几点上下班"
- 打卡记录决定"实际几点打卡"
- 计算比较两者差异得出迟到/早退/加班等结果

## 三、换班场景分析

### 3.1 典型场景

> 员工A（日常班，period=4，周轮换）和员工B（周六仓库值班，period=2，双周轮换）  
> 想在 **2026-06-15** 这天换班  
> - A 原来该上"日常"班（09:00-17:00）  
> - B 原来该上"周六仓库值班"班（09:00-12:00）  
> - 换班后：A 上 B 的班次，B 上 A 的班次

### 3.2 设计原则

换班本质上是 **两个员工的排班方案在指定日期上的临时交换**。不改变各自的永久排班方案，只在换班日期当天生效。

## 四、数据库模型设计

### 4.1 新增表

```sql
CREATE TABLE IF NOT EXISTS attendance_shift_swap_requests (
    id                   BIGSERIAL       PRIMARY KEY,
    applicant_emp_id     BIGINT          NOT NULL,     -- 申请人(考勤员工ID)
    target_emp_id        BIGINT          NOT NULL,     -- 换班对象(考勤员工ID)
    swap_date            DATE            NOT NULL,     -- 换班日期

    -- 申请人原排班信息(申请时的快照)
    applicant_orig_class_id     BIGINT,     -- 申请人原班次ID
    applicant_orig_period_no    INT,        -- 申请人原周期序号

    -- 申请人换班后排班信息(即对方的原排班)
    applicant_swap_class_id     BIGINT,     -- 申请人换班后班次ID
    applicant_swap_period_no    INT,        -- 申请人换班后周期序号

    -- 对方原排班信息(申请时的快照)
    target_orig_class_id        BIGINT,     -- 对方原班次ID
    target_orig_period_no       INT,        -- 对方原周期序号

    -- 对方换班后排班信息(即申请人的原排班)
    target_swap_class_id        BIGINT,     -- 对方换班后班次ID
    target_swap_period_no       INT,        -- 对方换班后周期序号

    reason               VARCHAR(500),               -- 换班原因
    status               VARCHAR(20)  NOT NULL DEFAULT 'pending',
                         -- pending(待审批) | approved(已批准)
                         -- rejected(已拒绝) | cancelled(已取消)
    approver_id          BIGINT,                     -- 审批人
    approve_at           TIMESTAMP,                  -- 审批时间
    approve_comment      VARCHAR(500),               -- 审批意见

    is_deleted           BOOLEAN DEFAULT FALSE,
    created_at           TIMESTAMP DEFAULT NOW(),
    created_by           BIGINT,
    updated_at           TIMESTAMP DEFAULT NOW(),
    updated_by           BIGINT
);

-- 索引
CREATE INDEX idx_shift_swap_date ON attendance_shift_swap_requests(swap_date);
CREATE INDEX idx_shift_swap_applicant ON attendance_shift_swap_requests(applicant_emp_id);
CREATE INDEX idx_shift_swap_target ON attendance_shift_swap_requests(target_emp_id);
CREATE INDEX idx_shift_swap_status ON attendance_shift_swap_requests(status);
```

### 4.2 设计说明

- 申请时快照双方排班信息，避免审批期间排班变更导致数据不一致
- 申请人和换班对象各存"原班次"和"换班后班次"——换班本质上是双方交换班次
- status 字段控制生命周期状态

## 五、业务流程设计

```
┌────────────────────────────────────────────────────────────┐
│ 1. 员工A发起换班申请                                         │
│    · 选择换班日期                                            │
│    · 选择换班对象B                                           │
│    · 填写原因                                                │
│    · 系统自动展示双方当日的排班信息                            │
│    · 提交后 status = pending                                 │
└─────────────────────────────────────────────────────┬──────┘
                                                      │
                                                      ▼
┌────────────────────────────────────────────────────────────┐
│ 2. 审批人(管理员)审批                                        │
│    · 查看换班详情：双方当前排班、换班后排班                    │
│    · 操作：批准 / 拒绝                                       │
│    · 批准后 status = approved                                │
│    · 拒绝后 status = rejected                                │
└─────────────────────────────────────────────────────┬──────┘
                                                      │
                                                      ▼
┌────────────────────────────────────────────────────────────┐
│ 3. 考勤计算时应用换班                                       │
│    · ScheduleOneDayAsync 中检查 approved 的换班申请          │
│    · 如果有，使用换班后的 class_id/period_no 排班            │
│    · 双方都按换班后的安排计算                                 │
└────────────────────────────────────────────────────────────┘
```

## 六、与现有考勤计算的集成方案

### 6.1 核心改动点

仅需要修改 `ScheduleOneDayAsync` 方法（在 `AttendanceCalculationService.cs` 中），在获取 `activeScheme` 之后、计算 `period2` 之前插入换班检查。

```csharp
ScheduleOneDayAsync(employeeId, scheduleDate, operatorId, existingRecordId)
{
    // 1. 获取生效排班方案（现有逻辑）
    var activeScheme = await _repo.GetEmployeeActiveSchemeAsync(employeeId, scheduleDate);
    if (activeScheme == null) return;

    // ★ 2. 新增：检查是否有已批准的换班申请
    var swap = await _repo.GetApprovedShiftSwapAsync(employeeId, scheduleDate);
    if (swap != null)
    {
        // 判断当前员工在换班中的角色，使用对应的换班后排班覆盖 activeScheme
        if (swap.ApplicantEmpId == employeeId)
        {
            activeScheme.ClassId = swap.ApplicantSwapClassId ?? activeScheme.ClassId;
            activeScheme.PeriodNo = swap.ApplicantSwapPeriodNo ?? activeScheme.PeriodNo;
        }
        else if (swap.TargetEmpId == employeeId)
        {
            activeScheme.ClassId = swap.TargetSwapClassId ?? activeScheme.ClassId;
            activeScheme.PeriodNo = swap.TargetSwapPeriodNo ?? activeScheme.PeriodNo;
        }
    }

    // 3. 后续逻辑不变...
    var schemeClass = await _repo.GetSchemeClassByIdAsync(activeScheme.ClassId);
    ...
}
```

### 6.2 新增仓储方法

```csharp
public async Task<AttendanceShiftSwapRequest?> GetApprovedShiftSwapAsync(
    long employeeId, DateTime date)
{
    using var conn = _connectionFactory.CreateConnection();
    return await conn.QueryFirstOrDefaultAsync<AttendanceShiftSwapRequest>(
        @"SELECT * FROM attendance_shift_swap_requests
          WHERE (applicant_emp_id = @EmpId OR target_emp_id = @EmpId)
            AND swap_date = @Date
            AND status = 'approved'
            AND is_deleted = FALSE
          LIMIT 1",
        new { EmpId = employeeId, Date = date });
}
```

### 6.3 不需要改动的逻辑

| 组件 | 不需要改动的原因 |
|------|----------------|
| ReCalculateOneDayAsync | 只依赖 attendance_record 中的 class_id/plan_id/b_time/e_time——已在排班阶段正确设置 |
| SaveAttendanceRecordAsync | 逻辑不变 |
| 考勤报表视图 | 数据来源不变 |
| 打卡记录合并逻辑 | 不变 |

## 八、边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 换班日期是节假日 | 节假日逻辑优先级最高（holiday 覆盖 planId/dayTypeId），换班不覆盖节假日 |
| 员工当天无生效排班 | 前端申请时提示"该员工当天无排班，无法换班"，阻止提交 |
| 同一天同一员工多个换班申请 | 只允许一个 approved 状态的申请生效 |
| 换班日期已过期 | 审批时检查日期，已过期的自动拒绝或标记过期 |
| 员工已离职 | 申请/审批时检查离职日期，离职后日期不可换班 |
| 换班双方同班次 | 可提交，但排班无变化，建议前端提示 |
| 换班后需重算考勤 | 审批通过后自动触发双方在换班日期的排班和重算 |
